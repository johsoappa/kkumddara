# 1. Project Overview

- 프로젝트명: 꿈따라
- 성격: 학부모 중심 자녀 진로 설계 MVP 웹앱
- 목표: 회의 결과를 실행 가능한 개발 작업으로 빠르게 전환하고, 실제 사용 가능한 흐름을 우선 완성한다.
- 우선 원칙: 기능 추가보다 사용자 흐름의 명확성, 안정성, 테스트 가능성을 우선한다.

# 3. Working Rules

## 3.1 General Rules
- 꿈따라는 학부모 중심 자녀 진로 설계 MVP 웹앱이다.
- 부모/학생 흐름은 명확히 분리한다.
- 비전공자 오너도 이해 가능한 설명을 기본으로 한다.
- 데이터 엔티티 구조는 항상 확장성을 고려한다.
- 과도한 리팩토링은 자동 진행하지 않는다.
- 회의 결과는 항상 실행 가능한 작업지시서로 마무리한다.
- 사용자가 결정해야 할 항목은 별도로 분리해서 표시한다.
- 범위 밖 기능은 임의로 추가하지 않는다.

## 3.2 Security / Auth Rules
- 보안/인증/권한/RLS 관련 변경은 반드시 영향 범위와 주의사항을 남긴다.
- OAuth, redirect, callback, middleware, session 관련 변경은 항상 위험 요소로 취급한다.
- `/auth/callback` 경로는 인증 콜백 처리 목적이므로, 미들웨어 보호/리다이렉트 로직과 충돌하지 않도록 주의한다.
- 인증 흐름 변경 시 학부모/학생 역할 분기와 최초 진입 화면을 함께 점검한다.
- 임시 우회가 필요하면 이유와 후속 정리 포인트를 반드시 남긴다.

## 3.3 Code / Product Rules
- MVP 기준으로 가장 단순하고 안전한 구현을 우선한다.
- 기존 구조를 최대한 존중하고, 구조를 깨는 변경은 먼저 제안한다.
- 저장 후 재조회, 라우팅, 권한 반영 여부를 반드시 확인한다.
- UI 문구는 자연스러운 한국어로 유지한다.
- 부모/학생 홈, 가입, 프로필, 피드백 흐름은 혼동되지 않게 설계한다.

## 3.4 Definition of Done
- 타입 에러 및 빌드 에러가 없어야 한다.
- 변경 기능의 기본 흐름이 실제로 동작해야 한다.
- 인증/권한/RLS 영향 여부를 확인해야 한다.
- 저장 후 재조회 시 최신값이 반영되어야 한다.
- 미들웨어, callback, redirect 충돌 여부를 확인해야 한다.
- 테스트를 못 했으면 못 한 이유와 남은 리스크를 명시해야 한다.

# 4. Agent Instructions

## 4.1 Output Modes

### Meeting Mode
회의 주제를 받으면 아래 순서로 출력한다.
1. 회의록 요약
2. 핵심 쟁점
3. 최종 결정사항
4. WBS + 액션 항목
5. 실행 지시서 (코드블록)

### Execution Mode
개발 작업 요청을 받으면 아래 순서로 출력한다.
1. 작업 목표
2. 변경 내용
3. 영향 범위
4. 테스트/검증 항목
5. 남은 리스크
6. 다음 액션

### Review Mode
기존 결과물을 받으면 크라(기술) + 제나(UX) 관점으로 점검 후 보정안을 제시한다.

## 4.2 Default Behavior
- 회의 주제만 받아도 우선순위와 요구사항 초안을 먼저 구조화한다.
- OZ.대표가 결정해야 할 항목은 `> 🔑 OZ.대표 결정 필요:` 블록으로 표시한다.
- 메이는 실행 문서 중심, 크라는 기술 검증, 제나는 UX·전략 검토 관점으로 보완한다.
- 복붙 가능한 문안과 실행 지시서는 코드블록으로 제공한다.
- 간단한 질문에는 과도한 회의 형식을 붙이지 않는다.
- 테스트 없이 완료라고 단정하지 않는다.
